Module 11: Amazon Simple Storage Service (S3)

11.1 What is S3?

Amazon Simple Storage Service (Amazon S3) is an object storage service that offers industry-leading scalability, data availability, security, and performance. Customers of all sizes and industries can use S3 to store and protect any amount of data for a range of use cases, such as websites, mobile applications, backup and restore, archive, enterprise applications, IoT devices, and big data analytics.

Key Characteristics:

  • Object Storage: Stores data as objects within buckets. Objects consist of data, metadata, and a key.
  • Highly Durable: Designed for 99.999999999% (11 nines) of durability, meaning your data is highly protected against loss.
  • Highly Available: Designed for high availability.
  • Scalable: Virtually unlimited storage capacity.
  • Secure: Robust security features, including encryption and access control.
  • Cost-Effective: Pay-as-you-go pricing with various storage classes.

11.2 S3 Buckets

An S3 bucket is a container for objects stored in Amazon S3. Every object is stored in a bucket. Buckets are associated with an AWS account and a specific AWS Region.

Bucket Naming Rules:

  • Must be globally unique across all AWS accounts.
  • Can contain lowercase letters, numbers, and hyphens.
  • Must be between 3 and 63 characters long.
  • Cannot be formatted as an IP address.

Bucket Properties:

  • Versioning: Keeps multiple versions of an object.
  • Logging: Records access requests to your bucket.
  • Static Website Hosting: Can host static websites directly from an S3 bucket.
  • Encryption: Configure default encryption for objects.

11.3 S3 Policies (Bucket Policies & ACLs)

S3 provides several mechanisms to control access to your buckets and objects.

Bucket Policies:

JSON-based policies attached to a bucket that specify who can access the bucket and its objects, and what actions they can perform. They are powerful and recommended for most access control scenarios.


{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "PublicReadGetObject",
      "Effect": "Allow",
      "Principal": "*",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::your-bucket-name/*"
    }
  ]
}

This example grants public read access to all objects in your-bucket-name.

Access Control Lists (ACLs):

A legacy access control mechanism for S3. ACLs define which AWS accounts or groups are granted access and the type of access. While still supported, bucket policies are generally preferred for their flexibility and centralized management.

Best Practice: Use IAM policies for user/role access and bucket policies for broader, resource-based access control. Disable public access to S3 buckets unless absolutely necessary and explicitly configured.

11.4 S3 Encryption

S3 offers various options to encrypt your data at rest and in transit.

Encryption at Rest:

  • SSE-S3 (Server-Side Encryption with S3-managed keys): AWS manages the encryption keys. Simplest option.
  • SSE-KMS (Server-Side Encryption with KMS-managed keys): AWS Key Management Service (KMS) manages the encryption keys. Provides more control and auditability.
  • SSE-C (Server-Side Encryption with Customer-provided keys): You provide and manage the encryption keys.
  • Client-Side Encryption: You encrypt the data before uploading it to S3.

Encryption in Transit:

Use HTTPS (SSL/TLS) endpoints to encrypt data in transit between your application and S3.

11.5 S3 Versioning

S3 Versioning allows you to keep multiple versions of an object in the same bucket. This protects against accidental deletions, overwrites, and can be used for data retention and compliance.

  • When versioning is enabled, every time an object is uploaded or modified, a new version is created.
  • Each version has a unique version ID.
  • Deleting an object with versioning enabled creates a delete marker, but the object versions remain.
  • You can suspend versioning, but it cannot be disabled once enabled.

11.6 S3 Lifecycle Management

S3 Lifecycle Management allows you to define rules to automatically transition objects to different storage classes or expire them after a certain period. This helps optimize storage costs and manage data retention.

Common Lifecycle Actions:

  • Transition: Move objects from one S3 storage class to another (e.g., from S3 Standard to S3 Standard-IA, then to S3 Glacier).
  • Expiration: Permanently delete objects after a specified time.

11.7 S3 Static Website Hosting

You can host a static website directly from an S3 bucket. This is a cost-effective and highly available way to host static content (HTML, CSS, JavaScript, images).

Steps:

  1. Create an S3 bucket with the same name as your domain (e.g., www.example.com).
  2. Enable static website hosting on the bucket.
  3. Configure a bucket policy to allow public read access to your objects.
  4. Upload your website files (index.html, error.html, CSS, JS, images).
  5. Configure your domain's DNS (e.g., using Route 53) to point to your S3 website endpoint.

Note: S3 static website hosting does not support server-side scripting (e.g., PHP, ASP.NET) or database connections. For dynamic websites, you would typically use EC2, Lambda, or other compute services.

11.8 S3 Pre-Signed URLs

An S3 pre-signed URL is a URL that you can generate to grant temporary access to a specific S3 object. This allows users who do not have AWS credentials or permissions to access a private object for a limited time.

Use Cases:

  • Allowing a user to upload a file to a private S3 bucket without giving them AWS credentials.
  • Allowing a user to download a private file from S3 without making the bucket public.
  • Distributing temporary access links for sensitive documents.

Pre-signed URLs are generated using your AWS credentials and specify the object key, HTTP method (GET, PUT), and an expiration time.
