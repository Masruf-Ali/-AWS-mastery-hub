Module 9: EC2 Storage Options

9.1 Amazon Elastic Block Store (EBS)

Amazon EBS provides persistent block storage volumes for use with Amazon EC2 instances. EBS volumes are highly available and reliable storage volumes that can be attached to any running instance in the same Availability Zone.

Key Characteristics:

  • Persistent: Data persists independently of the life of the EC2 instance.
  • Block Storage: Functions like a raw, unformatted hard drive. You can format it with a file system and mount it to your EC2 instance.
  • Network-Attached: EBS volumes are network-attached storage, not physically part of the instance.
  • Availability Zone Specific: An EBS volume can only be attached to an EC2 instance in the same Availability Zone.
  • Encryption: EBS volumes can be encrypted at rest and in transit.

EBS Volume Types:

  • General Purpose SSD (gp2/gp3): Balances price and performance for a wide variety of transactional workloads. Recommended for most workloads.
  • Provisioned IOPS SSD (io1/io2): Designed for I/O-intensive applications like large relational or NoSQL databases that require sustained IOPS performance.
  • Throughput Optimized HDD (st1): Low-cost HDD volume designed for frequently accessed, throughput-intensive workloads such as big data, data warehouses, and log processing.
  • Cold HDD (sc1): Lowest cost HDD volume designed for less frequently accessed workloads.
  • Magnetic (Standard): Previous generation, generally not recommended for new use cases.

9.2 Amazon Elastic File System (EFS)

Amazon EFS provides simple, scalable, elastic file storage for use with AWS Cloud services and on-premises resources. It is a fully managed service that can be accessed by multiple EC2 instances concurrently.

Key Characteristics:

  • File Storage: Provides a file system interface (NFSv4.1 and NFSv4.0 protocol).
  • Shared Access: Multiple EC2 instances can access the same EFS file system simultaneously.
  • Scalable: Automatically scales storage capacity up or down as you add or remove files.
  • Regional Service: EFS is a regional service, meaning it can be accessed from instances across multiple Availability Zones within the same Region.
  • Use Cases: Content management, web serving, data sharing, development environments, home directories.

9.3 EBS Snapshots

An EBS snapshot is a point-in-time backup of your EBS volume. Snapshots are incremental backups, meaning only the blocks that have changed since the last snapshot are saved. This makes them cost-effective and efficient.

Key Features:

  • Incremental: Only changed blocks are stored.
  • Stored in S3: Snapshots are stored in Amazon S3, providing high durability.
  • Cross-AZ/Region Copy: Snapshots can be copied across Availability Zones and Regions.
  • Restore: You can restore an EBS volume from a snapshot, creating a new volume.
  • AMI Creation: Snapshots are used in the process of creating Amazon Machine Images (AMIs).

Best Practice: Regularly take snapshots of your EBS volumes for disaster recovery and data protection.

9.4 Amazon Machine Images (AMIs)

An Amazon Machine Image (AMI) provides the information required to launch an instance. You must specify an AMI when you launch an instance.

An AMI includes:

  • A template for the root volume for the instance (e.g., an operating system, application server, and applications).
  • Launch permissions that control which AWS accounts can use the AMI to launch instances.
  • A block device mapping that specifies the volumes to attach to the instance when it's launched.

Types of AMIs:

  • Community AMIs: Publicly available AMIs provided by AWS or other users.
  • AWS Marketplace AMIs: AMIs with pre-installed software, often with associated costs.
  • My AMIs: AMIs you create from your own EC2 instances or snapshots.

Creating your own AMIs allows you to pre-configure instances with your applications and settings, speeding up deployment and ensuring consistency.

9.5 Instance Store

An instance store provides temporary block-level storage for your instance. This storage is located on disks that are physically attached to the host computer. Instance store is ideal for temporary storage of information that changes frequently, such as buffers, caches, scratch data, and other temporary content, or for data that is replicated across a fleet of instances, such as a load-balanced pool of web servers.

Key Characteristics:

  • Temporary: Data on instance store volumes persists only during the life of the instance. If the instance stops, hibernates, or terminates, the data is lost.
  • High Performance: Provides very high I/O performance, often using NVMe SSDs.
  • Physically Attached: Storage is physically attached to the host server.
  • Use Cases: Caching, scratch data, temporary storage for databases, buffers.

Important: Do not use instance store for data that needs to persist beyond the life of the instance.
