Module 10: Load Balancing & Auto Scaling

10.1 What is Load Balancing?

Load balancing is the process of distributing network traffic across multiple servers. This ensures that no single server is overloaded, improving application responsiveness and availability. An Elastic Load Balancer (ELB) automatically distributes incoming application traffic across multiple targets, such as EC2 instances, containers, IP addresses, and Lambda functions.

Benefits of Load Balancing:

  • High Availability: Distributes traffic across healthy targets, routing away from unhealthy ones.
  • Fault Tolerance: If one server fails, traffic is automatically redirected to other healthy servers.
  • Scalability: Allows you to add or remove servers to handle changes in traffic.
  • Security: Can integrate with AWS WAF and SSL/TLS termination.

AWS offers three types of Elastic Load Balancers:

  • Classic Load Balancer (CLB) - legacy, not recommended for new applications.
  • Application Load Balancer (ALB) - best for HTTP/HTTPS traffic.
  • Network Load Balancer (NLB) - best for extreme performance and static IP addresses.

10.2 Application Load Balancer (ALB)

Application Load Balancer (ALB) operates at the application layer (Layer 7) of the OSI model. It is ideal for HTTP and HTTPS traffic, providing advanced request routing targeted at the delivery of modern application architectures, including microservices and containers.

Key Features:

  • Path-based routing: Routes requests based on the URL path (e.g., /users to one target group, /products to another).
  • Host-based routing: Routes requests based on the host header (e.g., api.example.com to one target group, web.example.com to another).
  • Query string parameter routing: Routes based on query parameters.
  • HTTP header routing: Routes based on specific HTTP headers.
  • Support for containers and microservices: Can route to multiple ports on the same EC2 instance.
  • SSL/TLS termination: Offloads encryption/decryption from your instances.
  • Integration with AWS WAF: Provides web application firewall capabilities.

10.3 Network Load Balancer (NLB)

Network Load Balancer (NLB) operates at the connection level (Layer 4) of the OSI model. It is designed to handle millions of requests per second with ultra-low latency, making it ideal for extreme performance and static IP addresses.

Key Features:

  • High performance: Capable of handling volatile workloads and millions of requests per second.
  • Static IP addresses: Provides a static IP address per Availability Zone.
  • Preserves client IP: The client's IP address is preserved, allowing backend applications to see the original client IP.
  • TCP, UDP, and TLS traffic: Supports a wide range of protocols.
  • Ideal for: Gaming, IoT, high-performance computing, and applications that require static IP addresses.

10.4 Classic Load Balancer (CLB)

Classic Load Balancer (CLB) is the oldest generation of Elastic Load Balancers. It operates at both the request level (Layer 7) and connection level (Layer 4). While still supported, AWS recommends using ALB or NLB for new applications due to their advanced features and better performance.

Key Features:

  • Basic load balancing: Distributes traffic based on round-robin or least outstanding requests.
  • Supports HTTP, HTTPS, TCP, SSL.
  • Not recommended for new applications.

10.5 Auto Scaling

AWS Auto Scaling monitors your applications and automatically adjusts capacity to maintain steady, predictable performance at the lowest possible cost. It helps ensure that you have the correct number of EC2 instances available to handle the load for your application.

Key Components:

  • Launch Configuration/Template: A template that an Auto Scaling group uses to launch EC2 instances. It specifies the AMI ID, instance type, key pair, security groups, and other parameters.
  • Auto Scaling Group (ASG): A collection of EC2 instances that are treated as a logical grouping for the purposes of automatic scaling and management. An ASG defines the minimum, maximum, and desired number of instances.
  • Scaling Policies: Define when and how the ASG should scale.
    • Simple Scaling: Adjusts capacity based on a single CloudWatch alarm.
    • Step Scaling: Adjusts capacity based on a set of scaling adjustments, in response to a CloudWatch alarm.
    • Target Tracking Scaling: Most common and recommended. Adjusts capacity to maintain a specified target utilization level for a metric (e.g., keep CPU utilization at 50%).
    • Scheduled Scaling: Scales based on a schedule (e.g., increase capacity every weekday morning).

Benefits of Auto Scaling:

  • Fault Tolerance: Replaces unhealthy instances automatically.
  • High Availability: Ensures your application has enough capacity.
  • Cost Management: Scales down during low demand, saving costs.
  • Improved Performance: Scales up during high demand to maintain performance.

10.6 High Availability (HA) Design with ELB and Auto Scaling

Combining Elastic Load Balancing with Auto Scaling is a fundamental pattern for building highly available and scalable applications on AWS.

Architecture:

  1. Users access your application through a DNS record (e.g., myapp.example.com) that points to your ELB.
  2. The ELB distributes incoming traffic across multiple EC2 instances in different Availability Zones.
  3. An Auto Scaling Group manages the EC2 instances, ensuring that a healthy number of instances are always running.
  4. CloudWatch alarms trigger scaling policies to add or remove instances based on metrics like CPU utilization or network I/O.
  5. Instances are launched in different AZs to protect against single points of failure.

This setup provides a robust, self-healing, and cost-effective architecture for web applications and other scalable workloads.
