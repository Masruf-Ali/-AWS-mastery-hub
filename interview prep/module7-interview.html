Interview Prep: Module 7: IAM

Here are some common interview questions related to the IAM module, along with suggested answers.

1. What is AWS IAM, and why is it a crucial security service?

Answer: AWS IAM (Identity and Access Management) securely controls access to AWS resources. It's crucial because it allows you to manage who is authenticated (signed in) and authorized (has permissions) to use your resources. This enables fine-grained control, enforcing the principle of least privilege, which is fundamental to cloud security.

2. Explain the difference between an IAM User, Group, and Role.

Answer:

  • IAM User: Represents a person or application that interacts with AWS. It has long-term credentials (password, access keys).
  • IAM Group: A collection of IAM users. Policies are attached to groups, and users inherit those permissions, simplifying management. Groups do not have credentials.
  • IAM Role: An identity with specific permissions, but no long-term credentials. Roles are assumed by trusted entities (users, services, federated identities) to gain temporary permissions. They are used for delegating access.

3. What is the principle of least privilege in IAM, and why is it important?

Answer: The principle of least privilege dictates that an entity (user, group, role) should only be granted the minimum permissions necessary to perform its specific task and nothing more. It's important because it reduces the attack surface, limits the potential impact of a security breach, and prevents unauthorized actions, thereby enhancing the overall security posture of your AWS environment.

4. How do IAM Policies work? Describe their key components.

Answer: IAM Policies are JSON objects that define permissions. Key components include: Version (policy language version), Statement (an array of individual permission statements), Effect (Allow or Deny, Deny always overrides), Action (specific API calls allowed/denied, e.g., s3:GetObject), and Resource (the specific AWS resources, by ARN, to which the action applies). Policies can be identity-based (attached to users/roles) or resource-based (attached to resources like S3 buckets).

5. When would you use an IAM Role versus an IAM User? Provide a practical example.

Answer: You use an IAM User for a human or application that needs consistent, long-term access, like an administrator or a CI/CD tool. You use an IAM Role for temporary delegated access. For example, if an EC2 instance needs to read from an S3 bucket, you would attach an IAM Role with S3 read permissions to the EC2 instance. The instance assumes the role, gets temporary credentials, and can then access S3 without storing long-lived access keys on the instance.

6. What is Multi-Factor Authentication (MFA), and why is it recommended for AWS accounts?

Answer: MFA adds an extra layer of security by requiring more than just a password to sign in. It typically involves something you 'know' (password) and something you 'have' (MFA device code). It's highly recommended for AWS accounts, especially the root account and administrative IAM users, because it significantly reduces the risk of unauthorized access even if a password is compromised.

7. Explain how the 'Deny' effect works in an IAM policy.

Answer: The 'Deny' effect in an IAM policy is a powerful security tool because it always overrides an 'Allow' effect. If an action is explicitly denied by any policy, the principal will not be able to perform that action, even if another policy explicitly grants permission. This ensures that you can enforce strict prohibitions regardless of other permissions.

8. What are Access Keys in IAM, and what are the best practices for handling them?

Answer: Access Keys consist of an Access Key ID and a Secret Access Key. They are used for programmatic access to AWS (CLI, SDKs, direct API calls). Best practices include: never embedding them directly in code, rotating them regularly, giving them only the minimum necessary permissions (least privilege), and deleting unused access keys. IAM Roles providing temporary credentials are often preferred over long-lived access keys for applications.

9. How would you ensure a new IAM user only has read-only access to S3?

Answer: I would create an IAM user and then attach the AWS managed policy AmazonS3ReadOnlyAccess to either the user directly or, preferably, to an IAM Group that the user is a member of. This policy grants read-only permissions to all S3 buckets. If more granular access is needed (e.g., read-only to a specific bucket), I would create a custom IAM policy and attach it.

10. Describe a scenario where you would need to use a resource-based policy instead of an identity-based policy.

Answer: A common scenario is controlling access to an S3 bucket. While identity-based policies (attached to users/roles) can grant access, a Bucket Policy (a type of resource-based policy) defines who can access the bucket directly, even entities outside your AWS account. For instance, to allow public read access to a static website hosted on S3, you would use a bucket policy that grants `s3:GetObject` permission to the principal `*` (anonymous users) on the specific bucket resources.
